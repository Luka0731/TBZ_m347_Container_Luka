# KN03



## A) Eigenes Netzwerk
### Netzwerke und Container starten
```bash
docker network create --subnet=172.18.0.0/16 m347-net
```
Default bridge (busybox1 & busybox2):
```bash
docker run -dit --name busybox1 busybox
docker run -dit --name busybox2 busybox
```
m347-net (busybox3 & busybox4):
```bash
docker run -dit --name busybox3 --network m347-net --ip 172.18.0.2 busybox
docker run -dit --name busybox4 --network m347-net --ip 172.18.0.3 busybox
```

### Analyse
#### Default-Gateway
busybox1 und busybox2 teilen denselben Default-Gateway (172.17.0.1), da sie beide im default bridge Netzwerk sind. busybox3 und busybox4 teilen den Gateway des m347-net Netzwerks (172.18.0.1).
#### Gemeinsamkeiten
Container im gleichen Netzwerk können sich gegenseitig per IP anpingen. Innerhalb des user-defined Netzwerks (tbz) funktioniert zusätzlich die DNS-Auflösung per Containername.
#### Unterschiede
busybox1 kann busybox3 nicht per Name anpingen, weil sie in unterschiedlichen Netzwerken sind. Der Ping per IP schlägt ebenfalls fehl, da die Netzwerke voneinander isoliert sind.
#### Schlussfolgerung
Docker isoliert Netzwerke voneinander. Container können nur mit anderen Containern im gleichen Netzwerk kommunizieren. User-defined Netzwerke haben gegenüber dem default bridge den Vorteil, dass DNS-Auflösung per Containername funktioniert.
#### KN02 Betrachtung
In KN02 befanden sich die beiden Container im default bridge Netzwerk. Die Kommunikation funktionierte, weil sie im selben Netzwerksegment lagen. Mit dem `--link`-Flag wurde die Namensauflösung manuell konfiguriert, da das default bridge Netzwerk kein automatisches DNS bietet.
